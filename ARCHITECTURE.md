# Архітектура системи CryptoPortfolio Monitor

## 1. Тип застосунку та архітектурний підхід

Застосунок побудований за **веб-орієнтованою клієнт-серверною архітектурою** на базі фреймворку **Next.js 14 (App Router)**.

Це означає що сервер і клієнт об'єднані в одному проекті, але чітко розділені логічно:

- **Server Components** — React-компоненти що рендеряться виключно на сервері. Вони напряму звертаються до БД і зовнішніх API, не потрапляють у клієнтський бандл. Більшість сторінок (market, wallet detail, dashboard) — серверні компоненти (`app/(dashboard)/*/page.tsx`).
- **Client Components** — позначені директивою `'use client'`, виконуються в браузері, мають доступ до стану та подій. Використовуються для інтерактивних елементів: графіки цін, форми, список токенів з фільтрацією (`components/market/price-chart.tsx`, `components/wallets/token-balance-list.tsx`).
- **API Routes** — серверні endpoint-и (`app/api/**`) що обробляють запити клієнта або зовнішніх сервісів (Vercel Cron, Telegram Webhook).

Такий підхід дозволяє виконувати важкі операції (запити до БД, виклики зовнішніх API) на сервері, не навантажуючи клієнт, і водночас мати реактивний інтерфейс там де він потрібен.

## 2. База даних

### Тип: PostgreSQL (реляційна)

Реляційна БД обрана тому що дані мають чіткі зв'язки: користувач → гаманці → токени → ціни → тригери → логи. Це природно виражається через таблиці і foreign keys. NoSQL ускладнив би запити — наприклад, "всі активні тригери з даними юзера" це один JOIN у SQL проти кількох round-trips у MongoDB.

### ORM: Prisma

- **Типобезпека** — генерує TypeScript-типи з `schema.prisma`, помилки у запитах видно на етапі збірки, не в рантаймі
- **Міграції як код** — `prisma migrate`, зміни схеми версіонуються в git
- **Сумісність з serverless** — Prisma Client є синглтоном (`lib/db/prisma.ts`), не тримає зайвих з'єднань

Альтернативи: Drizzle (легший, менша екосистема), сирий SQL (більше контролю, але без типів і міграцій з коробки).

Основні таблиці: `User`, `Wallet`, `TokenBalance`, `TokenPrice`, `PriceHistory`, `PriceTrigger`, `NotificationLog`, `PortfolioSnapshot`.

## 3. Оновлення та підтягування цін

Ціни підтримуються актуальними через **три механізми**:

### 3.1 Cron-задача (автоматичне оновлення)
Файл: `lib/cron/price-updater.ts`, endpoint: `app/api/cron/update-prices/route.ts`

Запускається через **Vercel Cron** за розкладом. Pipeline складається з 5 кроків:
1. Збір унікальних token ID з активних тригерів і балансів гаманців
2. Запит актуальних цін у CoinGecko через `/coins/markets` (батчами по 250 монет)
3. Перерахунок USD-вартості балансів усіх користувачів
4. Збереження portfolio snapshot (знімок сукупної вартості портфоліо)
5. Перевірка тригерів і надсилання сповіщень у Telegram

### 3.2 Ручний Sync гаманця
Файл: `lib/services/wallet-sync.ts`, endpoint: `app/api/wallets/[id]/sync/route.ts`

Запускається вручну користувачем. Отримує токени з **Ankr API** (один multichain-запит для всіх EVM-мереж), потім збагачує ціни через **price-feed** (`lib/services/price-feed.ts`):
- Нативні монети (ETH, BNB, SOL) → **Binance** REST API
- ERC-20 / SPL токени → **DexScreener** за contract address
- Fallback для токенів без DEX-пари → **CoinGecko**

### 3.3 Кешування в TokenPrice
Таблиця `TokenPrice` є внутрішнім кешем цін. Заповнюється при завантаженні сторінки Market та при cron-оновленні. Wallet sync зчитує ціни з кешу перш ніж робити зовнішні запити — це економить API-квоти.

## 4. Пошук

Реалізований **повнотекстовий пошук на стороні сервера** через CoinGecko API.

Файл: `lib/services/coingecko.ts` (функція `searchCoins`), endpoint: `app/api/market/search/route.ts`

**Принцип роботи:**
- Клієнт відправляє GET-запит на `/api/market/search?q={query}`
- Сервер викликає CoinGecko `/search?query=...` і повертає список монет з їх `id`, `symbol`, `name`, `thumb`, `marketCapRank`
- Результати не кешуються (пошук має бути актуальним)

**Тип пошуку:** нечіткий (fuzzy) пошук на стороні CoinGecko по назві та символу монети. Локально ми лише фільтруємо результати (точний збіг за символом, сортування за market cap rank) — це реалізовано у `components/wallets/token-balance-list.tsx` (функція `handleSearchMarket`) та у `lib/services/wallet-sync.ts` (функція `enrichMissingLogos`).

**Компонент пошуку** (`components/market/token-search.tsx`) використовує debounce — запит відправляється не при кожному символі, а після паузи у введенні.

## 5. Захист даних та контроль доступу

### Аутентифікація
Використовується **NextAuth v5** (Auth.js). Конфігурація: `lib/api/auth-guard.ts`.

Підтримує credentials-провайдер (email + пароль). Паролі зберігаються у хешованому вигляді через **bcrypt** (`api/auth/register/route.ts`). Сесія реалізована через **JWT-токен** у httpOnly cookie — недоступний для JavaScript на клієнті, що захищає від XSS.

### Авторизація запитів
Кожен захищений API endpoint починається з:
```
const guard = await requireUser();
if (!guard.ok) return guard.response;
```
Функція `requireUser` (`lib/api/auth-guard.ts`) перевіряє JWT-сесію і повертає поточного користувача. Якщо сесія відсутня — повертає 401.

Дані ізольовані між користувачами: всі запити до БД фільтруються за `userId` (наприклад, `where: { userId: guard.user.id }`).

### Захист Cron endpoint
Endpoint `app/api/cron/update-prices` перевіряє `Authorization: Bearer {CRON_SECRET}` або заголовок `x-vercel-cron: 1`. Без цього — повертає 401, навіть у production.

### Захист Telegram Webhook
Запити від Telegram перевіряються за токеном бота через `TELEGRAM_BOT_TOKEN` у змінних середовища.

### Захист від ін'єкцій

**SQL-ін'єкції → Prisma.** Всі запити до БД параметризовані автоматично. `$queryRaw` з підстановкою рядків у коді відсутній.

**Валідація вхідних даних → Zod** (`lib/utils/validators.ts`). Кожен API-ендпоінт валідує body через `safeParse` перед використанням. Якщо дані не відповідають схемі — одразу `400`, до БД не доходить:
- `email` — формат через `z.string().email()`
- `password` — мін 8, макс 100 символів
- `address` — regex для EVM (`/^0x[a-fA-F0-9]{40}$/`) і Solana (`/^[1-9A-HJ-NP-Za-km-z]{32,44}$/`)
- числові поля (`threshold`, `interval`, `targetPrice`) — тип + діапазон (`min/max`)
- `telegramChatId` — тільки цифри (`/^-?\d+$/`)
- `days` для графіку — тільки з білого списку `[1, 7, 30, 90, 365]`

**XSS → React JSX.** Весь вивід через `{variable}` екранується автоматично. `dangerouslySetInnerHTML` у коді відсутній.

**Ізоляція даних.** Кожен запит до БД фільтрується за `userId` з перевіреної сесії — неможливо отримати чужі дані навіть знаючи ID ресурсу.

## 6. API та запити

### Внутрішні API Routes (Next.js Route Handlers)
| Endpoint | Метод | Призначення |
|---|---|---|
| `/api/auth/[...nextauth]` | GET/POST | NextAuth сесії, логін, логаут |
| `/api/wallets` | GET/POST | Список і створення гаманців |
| `/api/wallets/[id]/sync` | POST | Синхронізація гаманця |
| `/api/wallets/[id]/tokens` | GET | Токени гаманця |
| `/api/market/prices` | GET | Топ-100 монет з CoinGecko |
| `/api/market/search` | GET | Пошук монет |
| `/api/market/[tokenId]` | GET | Деталі токена |
| `/api/market/[tokenId]/history` | GET | Графік ціни |
| `/api/alerts` | GET/POST | Тригери користувача |
| `/api/alerts/[id]` | PATCH/DELETE | Редагування тригера |
| `/api/alerts/logs` | GET | Лог сповіщень |
| `/api/portfolio` | GET | Портфоліо |
| `/api/cron/update-prices` | GET/POST | Cron-оновлення цін |
| `/api/telegram/webhook` | POST | Telegram webhook |

### Зовнішні API (сторонні сервіси)
| Сервіс | Призначення | Ліміт (безкоштовно) |
|---|---|---|
| CoinGecko | Ціни, деталі, пошук монет | ~30 req/хв (demo key) |
| Ankr | Баланси EVM-гаманців (multichain) | 30M credits/місяць |
| Moralis | Баланси Solana + fallback EVM | 40K req/місяць |
| DexScreener | Ціни ERC-20 за contract address | 300 req/хв |
| Binance | Ціни нативних монет | Без ключа, без ліміту |
| Binance Futures | Відкритий інтерес | Без ключа |
| Bybit | Відкритий інтерес | Без ключа |
| OKX | Відкритий інтерес | Без ключа |
| Solana RPC | Транзакції Solana (Moralis не має ендпоінту) | Публічний, нестабільний |
| Telegram Bot API | Надсилання сповіщень | Без ліміту (до 30/сек) |

**Пріоритет цін:** Binance → DexScreener → CoinGecko (fallback)

**Обробка лімітів:** активного throttling немає. CoinGecko і Moralis — retry x3 з backoff (2→4→8с) при 429/5xx. DexScreener — конкурентність обмежена до 8 паралельних запитів. CoinGecko — батчинг по 250 токенів. При збої ціни залишаються старими до наступного тіку.

**Solana RPC** (`lib/services/moralis.ts`) — метод `getSignaturesForAddress`. Повертає тільки хеші і timestamp, без деталей транзакції (суми, адреси = `null`).

**Healthcheck** — `GET /api/health` повертає `{ status: 'ok', timestamp }`. БД і зовнішні API не перевіряються.

## 7. Тригери і Cron

### Cron
`vercel.json` — розклад `* * * * *` (щохвилини). Endpoint `GET /api/cron/update-prices` захищений `CRON_SECRET`.

### Тригери (PriceTrigger)
Логіка в `lib/cron/price-updater.ts`. Два типи:

**PERCENT** — спрацьовує коли ціна змінюється на `threshold`% за `interval` хвилин. `lastPrice` оновлюється після кожного тіку. Функція `evaluateTrigger` — чиста, без I/O.

**PRICE_TARGET** — спрацьовує **один раз** коли ціна перетинає вказаний рівень (cross-detection: `lastPrice` був по один бік від цілі, `currentPrice` — по інший). Після спрацювання тригер автоматично деактивується (`isActive: false`). `lastPrice` оновлюється щохвилини навіть якщо тригер ще не спрацював — щоб наступний тік міг виявити перетин.

При спрацюванні:
1. Надсилається повідомлення у Telegram
2. Записується рядок у `NotificationLog` зі статусом `sent` або `failed`
3. Оновлюється `lastPrice` і `lastCheckedAt`

## 8. Важливі питання на захисті

**Чому Next.js а не окремий backend?**
Next.js дозволяє поєднати frontend і backend в одному проекті з єдиним деплоєм. Server Components виконують запити до БД безпосередньо, без REST-прошарку між ними — це спрощує архітектуру і зменшує latency.

**Чому не WebSocket для реалтайм оновлення цін?**
Системі не потрібен справжній реалтайм — достатньо оновлення щохвилини через cron. WebSocket додав би складність інфраструктури (stateful-сервер, не serverless) без суттєвої користі.

**Як масштабується система?**
Завдяки архітектурі serverless (Vercel) кожен API-запит виконується незалежно. БД (PostgreSQL) масштабується вертикально, або можна підключити connection pooling (PgBouncer). Cron обмежений одним інстансом, що виключає паралельні запуски.

**Як обробляються помилки зовнішніх API?**
Всі зовнішні API мають retry-логіку з exponential backoff (3 спроби, затримки 2→4→8 секунд). Якщо API недоступний — повертається graceful fallback: кешовані дані або відображення помилки без краша сторінки. Реалізовано в `lib/services/coingecko.ts` (функція `withRetry`).

**Як зберігається безпека паролів?**
Паролі ніколи не зберігаються у відкритому вигляді. При реєстрації виконується bcrypt hash з rounds = 12. При логіні — порівняння через `bcrypt.compare`. Навіть якщо БД буде скомпрометована, паролі не можна отримати зворотньо.

**Що таке Server Components і навіщо вони?**
React Server Components (RSC) — компоненти що виконуються виключно на сервері. Вони можуть асинхронно звертатись до БД, не потрапляють у JavaScript-бандл клієнта, зменшують розмір завантаження. Сторінки дашборду, ринку, гаманців — всі серверні компоненти.

**Чому Prisma а не сирий SQL?**
Prisma забезпечує типобезпеку на рівні компілятора — неможливо написати запит з неіснуючим полем і не отримати помилку збірки. Це критично важливо у великих проектах. Крім того, Prisma migrate дозволяє версіонувати схему БД як код.

**Як захищено від SQL-ін'єкцій?**
Prisma використовує параметризовані запити автоматично — жодне значення не підставляється у SQL-рядок напряму. Це повністю виключає SQL-ін'єкції на рівні ORM.

**Як реалізована ізоляція даних між користувачами?**
Кожна операція з даними фільтрується за `userId` з перевіреної сесії (`guard.user.id`). Неможливо отримати дані іншого користувача навіть знаючи ID ресурсу — сервер завжди перевіряє що ресурс належить поточному користувачу.

**Де зберігаються секрети (API ключі)?**
Виключно у змінних середовища (`.env`), ніколи у коді. На Vercel — у розділі Environment Variables. Файл `.env` додано до `.gitignore`.
